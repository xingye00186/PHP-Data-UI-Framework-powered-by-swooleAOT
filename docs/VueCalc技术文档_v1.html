类 Vue 数据驱动的桌面计算器

一、项目概述

VueCalc 是一个基于 Swoole AOT 编译器编译的 Windows 桌面计算器应用。它采用 PHP/C++ 混合架构，实现了一套 类 Vue 的响应式数据驱动渲染模型（已针对 AOT 编译约束做了适配）。

核心思想：所有业务逻辑由 PHP 编写，C++ 仅提供最薄的 Win32 窗口/GDI 绘制原语层。UI 渲染完全由数据驱动——UI = f(state)。

关键 AOT 约束：Swoole Compiler AOT 不支持 __get/__set 魔术方法。因此本实现采用直接属性声明 + 手动脏标记 (dirty flag) 模式，而非标准 PHP 的魔术方法代理模式。

2.2 数据层：直接属性 + 手动脏标记（AOT 兼容模式）

Calculator 声明 6 个 public 属性作为响应式状态：

class Calculator extends ReactiveComponent
{
    public string $display = '0';      // 当前显示值
    public string $expression = '';    // 表达式（右上角小字）
    public string $operand1 = '';      // 第一个操作数
    public string $operator = '';      // 当前运算符
    public bool $newInput = true;      // 是否开始新输入
    public bool $hasDecimal = false;   // 是否已输入小数点
}

每个修改状态的方法末尾显式设置 $this->dirty = true：

public function inputDigit(string $digit): void
{
    if ($this->newInput) {
        $this->display = $digit;
        $this->newInput = false;
        $this->hasDecimal = false;
    } else {
        if ($this->display === '0' && $digit !== '.') {
            $this->display = $digit;
        } else {
            $this->display .= $digit;   // 直接拼接，不依赖 __get
        }
    }
    $this->dirty = true;                // ← AOT兼容：手动脏标记
}
为什么不用 __get/__set？
Swoole Compiler AOT 将 PHP 编译为 C++ 后，__get/__set 魔术方法不会被自动调用。例如 $this->display .= $digit 在标准 PHP 中会先触发 __get 读再 __set 写，但 AOT 编译后 __get 不被调用 → 返回 null → null .= '5' → TypeError 崩溃。

2.3 渲染层：脏标记驱动 → C++ GDI 绘制

// CalcApp::run() 主循环 — main.php
while ($running) {
    // 1️⃣ 消费所有 Windows 消息
    while (true) {
        $msg = vue_peek_message();
        if (count($msg) == 0) break;   // 队列空 → 退出内层
        // ... 处理 WM_LBUTTONDOWN / WM_QUIT ...
    }

    // 2️⃣ 脏检查 → 仅在数据变化时重绘
    if ($this->calc->dirty) {
        $this->renderer->render();     // 数据驱动渲染
        $this->calc->dirty = false;    // 清除标记
    }

    usleep(16000);  // ~60 FPS
}

五、关键设计决策与 Bug 修复经验

决策 / 修复原因
属性必须在子类声明为 publicAOT 不支持 __get/__set，未声明属性读写不会触发魔术方法
手动 dirty 标记替代 __set 中的自动标记，确保 AOT 兼容
脏标记而非每帧渲染仅在数据变化时重绘，空闲时 CPU 几乎为零
双缓冲 GDICreateCompatibleDC → 画完 → BitBlt，避免闪烁
PHP 逻辑 + C++ 原语业务代码保持 PHP 的灵活，C++ 仅负责 Win32 API 调用
避免 json_encode/json_decode可能不在 AOT 运行时环境中
避免 str_contains()（PHP 8.0）改用 strpos()，确保 PHP 7+ 兼容
内层消息循环 + 外层渲染先清空消息队列，再一次性重绘（批量更新）
Bug 修复清单
  1. any($value) → 移除。不是 PHP 标准函数，运行时 fatal error
  2. str_contains() → strpos()。PHP 8.0 函数，AOT 可能不支持
  3. __get/__set → 直接属性。AOT 编译后魔术方法不被调用，$this->display .= $digit 读回 null 导致 TypeError
  4. 静态属性类型：ChangeQueue $queue → ?ChangeQueue $queue = null，加空值检查

7.2 响应式实现：透明 vs 显式

Vue 3 的 Proxy 拦截（透明）

state.display = '7'     // Proxy set 拦截 → 触发依赖更新 → 组件重渲染
state.display += '5'    // Proxy get+set 拦截 → 自动追踪

// 每个修改方法末尾手动标记
public function inputDigit(string $digit): void {

// 渲染循环检测 dirty → render()

八、面向 AOT 的 .vue SFC 编译器设计路线图

最终目标：让 PHP 替代 JS 实现数据与业务逻辑，C++ 渲染界面。开发者只需编写 .vue 单文件组件，由 PHP 编写的 SFC 编译器将其编译为 PHP ReactiveComponent 子类 + C++ 原生布局数据，再经 AOT 编译为 exe。

8.4 现状：缺什么？

组件状态说明
Template Parser ❌ 缺失 需要 PHP 实现的 HTML-like 词法+语法解析器。可参考 PHP 内置 DOM 解析，或手写递归下降 parser
Layout Engine ❌ 缺失 需要 CSS Box Model（margin/border/padding）计算 + flex/grid 布局算法。挑战：GDI 没有布局引擎，坐标必须精确计算
Style → GDI Mapper ❌ 缺失 CSS 子集到 GDI 属性映射表。色彩空间需 BGR 转换
v-model 代码生成 ❌ 缺失 指令解析 + PHP 属性声明 + 读取/写入代码自动生成
@event 处理 ❌ 缺失 点击坐标 → 按钮路由代码自动生成（当前手动在 CalcApp 中硬编码）
ReactiveComponent 基类 ✅ 已有 dirty 标记 + 共享基础设施
GDI 渲染原语 ✅ 已有 C++ 层：窗口/双缓冲/文本/矩形/按钮/边框
消息泵 ✅ 已有 PeekMessage + DispatchMessage 循环
AOT 编译链 ✅ 已有 Swoole Compiler + MSVC → exe

8.5 可能的问题与改进方向

问题1：AOT 编译器的约束

  • 无反射：不能依赖运行时类名字符串创建对象。布局必须是编译期确定的
  • 无 eval/include：SFC 编译器必须在编译前运行（作为 AOT 的预处理步骤），生成的 PHP 代码作为普通源文件参与 AOT 编译
  • 无魔术方法：__get/__set/__call 不可靠，代码生成器必须生成显式的属性声明和 dirty 标记
  • 函数可用性不确定：需测试哪些 PHP 标准函数在 AOT 运行时可用，SFC 编译器生成的代码只能用确认可用的函数

问题2：CSS → GDI 映射的局限性

GDI 不是浏览器渲染引擎。以下 CSS 特性无法直接映射：

  • opacity / rgba() → GDI 不支持 alpha 通道（需用 GDI+ 或 Direct2D）
  • box-shadow → 需手动绘制多层矩形
  • border-radius → RoundRect 可用但效果有限
  • transform / 动画 → 需要额外的数学计算和帧插值
  • flex / grid → 需手动实现布局算法
改进方向：将 C++ 渲染层从 GDI 升级为 Direct2D 或 Skia。Direct2D 支持硬件加速、alpha 通道、抗锯齿；Skia（Chrome/VSCode 同款）则提供完整的 2D 图形能力。但这会显著增加 exe 体积和依赖复杂度。

问题3：布局计算的精度

Web 浏览器有完整的布局引擎（Blink/WebKit），而桌面端需要自己计算像素坐标。改进方案：

  1. 使用固定尺寸布局（简单可靠，牺牲灵活性）
  2. 集成轻量布局库（如 Facebook Yoga，C 语言实现，可嵌入）
  3. 在编译期由 PHP SFC 编译器计算好布局，生成静态坐标数据

问题4：组件树的表达

Vue 的组件树是动态创建的。在 AOT 约束下，组件树必须在编译期确定。改进方案：

// 编译期展开的组件树
class App extends ReactiveComponent {
    // 子组件作为成员变量声明（编译期确定类型）
    private HeaderComponent $header;
    private ContentComponent $content;
    private FooterComponent $footer;
    
    public function __construct() {
        $this->header  = new HeaderComponent();
        $this->content = new ContentComponent();
        $this->footer  = new FooterComponent();
    }
}

问题5：差异化重绘（增量渲染）

当前每次 dirty 都全量重绘整个界面。对于简单计算器够用，但对于复杂 UI 有性能问题。改进方案：

  • 给每个布局节点标记 dirty，只重绘变化的区域
  • SFC 编译器自动分析模板依赖，生成精准的脏标记代码
  • 例如：v-model:display 变化 → 只 markDirty('display_area')

8.6 实现路线图

  1. Phase 1（短期）：手写 PHP Template Parser —— 解析 .vue 的 <template> 块，输出 AST
  2. Phase 2（短期）：Style → GDI 映射表 —— CSS 属性子集到 GDI 常量的字典
  3. Phase 3（中期）：Layout Engine —— 基于 AST + 映射表 → 生成坐标布局数据
  4. Phase 4（中期）：Code Generator —— AST + 布局数据 → 生成 PHP 子类（属性+方法） + C++ 头文件（布局常量）
  5. Phase 5（长期）：v-model 指令解析 —— 自动生成属性声明、getter/setter、dirty 标记代码
  6. Phase 6（长期）：组件树支持 —— 嵌套组件编译展开
  7. Phase 7（远期）：增量渲染引擎 —— 节点级 dirty 标记，区域重绘
  8. Phase 8（远期）：渲染层升级 —— GDI → Direct2D，支持更丰富的视觉效果
