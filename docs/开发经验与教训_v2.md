## 一、项目背景

VueCalc v2 实现了完整 SFC 编译器管道：`Calculator.vue` → `sfc-compiler.php` → `.gen.php` → `AOT Compiler` → `vue_calc.exe` (232KB)。

技术栈：
- **PHP**：业务逻辑 + SFC 编译器 + 响应式框架 + 渲染调度
- **C++ (Win32 GDI)**：窗口管理、GDI 绘制原语
- **Swoole Compiler**：PHP → C++ AOT 编译为原生 exe

## 二、踩过的坑（按时间线排序）

### 2.1 `require_once` 在顶层导致 "All execution code must be within a function"

**问题**：AOT 编译器要求所有可执行代码必须在函数内部。

**解决**：移除所有顶层 `require_once`。AOT 通过 `project.yml` 的 `sources` 配置读取所有文件，编译期全量链接。

**教训**：AOT 编译 = 编译期全量链接。`project.yml` 的 `sources` 是唯一的文件依赖声明方式。

---

### 2.2 `__DIR__` 魔术常量不能用于属性默认值

**问题**：`__DIR__` 用于类属性的默认值时，AOT 编译器报错 "must be a global, non-magic constant"。

**解决**：移到构造函数中赋值。

---

### 2.3 `Swoole\Table` 和 `Swoole\Atomic` 在 AOT 环境中不可用

**解决**：重写为纯 PHP 实现——数组环形缓冲替代 `Swoole\Table`。

**教训**：AOT 运行时 ≠ PHP CLI 运行时。编译产物只包含 PHP 核心运行时，不含任何 PHP 扩展。

---

### 2.4 静态属性反射问题

**问题**：反射遍历时误把父类静态属性当响应式属性初始化。

**解决**：添加 `$prop->isStatic()` 过滤。

---

### 2.5 PHP 8.x 类型化属性拒绝 `ReactiveValue` 对象

**教训**：类型系统与动态代理不相容。在 AOT 约束下，`public` 无类型属性是最安全的选择。

---

### 2.6 `any($value)` —— 根本不是 PHP 函数

**解决**：移除 `any()`，直接存储值。

**教训**：LLM 输出的每个函数调用都必须确认是 PHP 内置或自定义函数。

---

### 2.7 `str_contains()` 是 PHP 8.0 函数

**解决**：替换为 `strpos($haystack, $needle) !== false`（PHP 4 兼容）。

---

### 2.8 `__get/__set` 魔术方法在 AOT 中不被调用 ⚠️ 最大坑

**问题**：`$this->display .= $digit` 在 AOT 编译后 `__get` 不被调用，返回 null → TypeError。

**解决**：彻底放弃魔术方法。Calculator 声明 6 个 `public` 属性，每个方法末尾手动 `$this->dirty = true`。

**教训**：
- AOT 编译器的行为 ≠ 标准 PHP 的行为
- `__get/__set` 不要指望——声明属性直接编译为 C++ 成员变量
- "优雅"要妥协于"可靠"——手动脏标记是 AOT 下唯一可靠的方式

---

### 2.9 静态属性类型声明

**解决**：`protected static ?ChangeQueue $queue = null;` + 空值检查。

---

### 2.10 SFC 编译器：样式正则分隔符冲突 🆕 v2

**问题**：
```php
// ❌ # 既是分隔符又是 CSS 颜色前缀
preg_match('#background\s*:\s*#([0-9a-fA-F]{6})#', $body, $m)
// 第二个 # 被当成分隔符结束 → "Unknown modifier '('"
```

**影响**：所有 CSS 颜色解析失败，按钮全部使用默认颜色（灰色）。

**解决**：改用 `~` 分隔符：
```php
preg_match('~background\s*:\s*#([0-9a-fA-F]{6})~', $body, $m)
```

**教训**：选择正则分隔符时要考虑模式内容中的特殊字符。对于 CSS 颜色值模式，`~` `@` `!` 等不常见的分隔符更安全。

---

### 2.11 SFC 编译器：文本元素未解析 🆕 v2

**问题**：
```php
// ❌ [a-z-]+ 不匹配 :bind 的冒号
preg_match_all('#<text\s+((?:[a-z-]+="[^"]*"\s*)+)\s*/>#', ...)
// 输出：Elements: 2 (rects only, texts missing)
```

**解决**：添加 `:` 到属性名字符类：
```php
preg_match_all('#<text\s+((?:[a-z:@-]+="[^"]*"\s*)+)\s*/>#', ...)
```

**教训**：正则字符类设计要考虑所有可能的属性名格式（包括 `:bind`、`@click` 等自定义指令）。

---

### 2.12 SFC 编译器：文件名点号导致 C++ 编译错误 🆕 v2

**问题**：文件名 `CalculatorLayout.gen.php` 的点号被 AOT 用于生成 C++ 函数名：
```cpp
// ❌ .gen 在 C++ 中不是合法标识符
static void register_CalculatorLayout.gen_symbols(int module_number)
// 被解析为：register_CalculatorLayout . gen_symbols(...)  → 语法错误
```

**解决**：改为下划线命名 `CalculatorLayout_gen.php`。

**教训**：AOT 编译器会将 PHP 文件名映射为 C++ 符号名。文件名中不要使用点号、连字符等 C++ 非法标识符字符。仅使用 `[a-zA-Z0-9_]`。

---

### 2.13 SFC 编译器：const 复杂数组不被 AOT 支持 🆕 v2

**问题**：
```php
// ❌ var_export 生成的嵌套数组不能作为 AOT 全局常量
const LAYOUT = [
    'elements' => array (0 => array ('type' => 'rect', ...), ...),
    'buttons'  => array (...),
];
```

AOT 编译器跳过 `LAYOUT` 常量注册，生成的文件中只有简单常量（`WINDOW_WIDTH`、`WINDOW_HEIGHT`）。

**解决**：改为函数返回：
```php
function getLayout(): array
{
    return [
        'elements' => [...],
        'buttons'  => [...],
    ];
}
```

**教训**：AOT 编译器对 `const` 的支持有限——简单标量常量（int、string）OK，复杂嵌套数组不支持。数据量大的配置用函数返回比全局常量更可靠。

---

### 2.14 分层渲染 z-order 覆盖不完整 🆕 v5 M3

**问题**：弹窗（AboutDialog）显示时，底层按钮残留在弹窗区域内，出现"半透明"的视觉穿透效果。

根因：`BaseRenderer.render()` 将元素和按钮分为**两个独立分层循环**——先遍历所有层画完全部元素，再遍历所有层画完全部按钮。这导致高层按钮会在高层元素之上，但低层按钮的绘制发生在高层元素之后，低层按钮残留在画面上无法被高层元素完整覆盖。

```php
// ❌ 两个独立循环：低层按钮画在高层元素之后 → z-order 错乱
foreach ($layers as $layer) { /* 只画元素 */ }
foreach ($layers as $layer) { /* 只画按钮 */ }
```

**解决**：合并为**单一循环**——每层内先画元素再画按钮，确保高层完整覆盖低层：

```php
// ✅ 单一循环：每层内先画元素再画按钮 → 高层完全覆盖低层
foreach ($layers as $layer) {
    // 先画该层的所有元素（背景、文本等）
    foreach ($layer['elements'] as $el) { ... }
    // 再画该层的所有按钮
    foreach ($layer['buttons'] as $btn) { ... }
}
```

**教训**：分层渲染的 z-order 正确性要求**同一层内所有内容作为一个整体**绘制。将不同类型的绘制对象拆成独立循环会破坏层间覆盖关系。每层应当是"一笔画完"——元素+按钮在同一次迭代中绘制。

---

### 2.15 vendor/autoload.php 硬编码绝对路径 🆕 v5 M3

**问题**：
```php
// ❌ 硬编码绝对路径——迁移环境后立刻失效
require_once 'F:/work/swoole_compiler/vendor/autoload.php';
```

项目整体迁移或在不同机器上构建时，硬编码的绝对路径导致 `autoload.php` 找不到，Composer 依赖加载失败。

**解决**：改用 `__DIR__` 相对路径：
```php
// ✅ 基于当前文件位置的相对路径——可移植
require_once __DIR__ . '/../swoole_compiler/vendor/autoload.php';
```

**教训**：PHP 代码中**永远不要硬编码绝对路径**。`__DIR__` 是最可靠的路径锚点——基于文件自身位置计算，不受工作目录、环境变量、机器差异影响。构建脚本 (`main_build.bat`) 同理，使用 `%~dp0` 而非硬编码路径。

---

### 2.16 构建脚本遗漏 SDK/lib/ 路径查找 php8embed.lib 🆕 v5 M3

**问题**：`main_build.bat` 在查找 `php8embed.lib` 时只搜索了 `lib/` 和 `lib/lib/` 两个路径，遗漏了 `SDK/lib/`。在标准 SDK 目录布局下，`php8embed.lib` 位于 `swoole_compiler/SDK/lib/php8embed.lib`，构建因找不到库文件而失败。

**解决**：在路径查找列表中添加 `SDK/lib/`：
```bat
rem ✅ 添加 SDK/lib/ 作为第三候选路径
if exist "%~dp0swoole_compiler\SDK\lib\php8embed.lib" (
    set PHP_LIB_DIR=%~dp0swoole_compiler\SDK\lib
)
```

**教训**：构建脚本的路径查找必须覆盖项目中**所有实际存在的目录布局**。添加新的目录结构后要同步更新构建脚本的搜索路径列表，否则会出现"明明文件在那里但构建找不到"的情况。

---

### 2.17 文本对齐功能缺失：不支持 center 🆕 v5 M3

**问题**：`BaseRenderer` 只实现了 `right` 文本对齐（用于数字显示右对齐），`center` 对齐完全缺失。AboutDialog 的标题和内容文本需要居中对齐，但编译后全部左对齐显示。

**解决**：在 `drawText()` 中添加 `center` 对齐分支：
```php
// ✅ 支持 center 对齐
if ($align === 'center') {
    $textX = $x + ($w - $textWidth) / 2;
} elseif ($align === 'right') {
    $textX = $x + $w - $textWidth;
} else {
    $textX = $x; // left（默认）
}
```

**教训**：渲染器的对齐功能应覆盖常见布局需求（left / center / right），不能"用到再加"——每个新增组件都可能需要之前未实现的对齐方式。在设计渲染原语时就应规划完整的对齐支持。

---

### 2.18 UI 布局尺寸与组件样式不匹配 🆕 v5 M3

**问题**：两个 UI 细节问题同时暴露：

1. **DisplayPanel** 显示文本区域 `container-w=320` 过宽，与右侧 `?` 按钮重叠，导致数字尾部被按钮遮挡。
2. **AboutDialog** Close 按钮尺寸 `60×28` 过小且使用 `btn-func`（灰色），视觉上不够醒目，不符合对话框关闭操作的交互惯例。

**解决**：
- DisplayPanel：`container-w` 从 320 缩小到 282，为 `?` 按钮留出空间
- AboutDialog：Close 按钮从 `60×28` 加大到 `100×34`，样式从 `btn-func`（灰色）改为 `btn-op`（橙色）

**教训**：UI 组件的尺寸和样式要在**实际运行中验证**——仅凭计算值设计布局容易遗漏元素间的重叠问题。对话框的操作按钮应足够醒目（尺寸+颜色），灰色小按钮不符合"关闭/确认"等关键操作的视觉层级。

## 三、架构演进的教训

### 3.1 v1 初始架构（失败）

```
Calculator.reactiveProperties()  →  ReactiveComponent.__get/__set  →  ReactiveValue 代理
    动态声明属性                           拦截所有读写                       值+版本号
                                  + ChangeQueue 变更队列
```

失败原因：`__get/__set` 在 AOT 中不可靠。

### 3.2 v1 最终架构（可行但硬编码）

```
Calculator.public 属性  →  手动 $this->dirty = true  →  主循环检测 dirty → render()
  6个 string/bool            每个修改状态的方法末尾            CalcApp::run() 每帧检查
```

### 3.3 v2 SFC 架构（编译期预处理）

```
Calculator.vue  →  sfc-compiler.php  →  Calculator.gen.php + CalculatorLayout_gen.php  →  AOT → exe
  SFC 单文件         编译期预处理           ReactiveComponent 子类 + 布局数据              原生 GUI
```

核心变化：布局数据从运行时硬编码提升为编译期生成。

## 四、事件循环的设计经验

### 4.1 最终形态

```php
while ($running) {
    // 内层：批量处理所有 Windows 消息（1次1条）
    while (true) {
        $msg = vue_peek_message();
        if (count($msg) == 0) break;
        if (WM_LBUTTONDOWN) { handleClick(); }
        if (WM_QUIT) { $running = false; }
    }
    // 外层：消息清空后检查 dirty → 渲染
    if ($this->calc->dirty) {
        $this->renderer->render();
        $this->calc->dirty = false;
    }
    usleep(16000);
}
```

### 4.2 v2 的 handleClick 改进

v1 使用行列计算（`intdiv` + `BUTTONS` 数组），v2 改为像素碰撞检测——遍历 `getLayout()['buttons']` 进行 bounds check。这更灵活，支持任意排列的按钮。

## 六、开发流程教训

### 6.1 调试方法

1. **`no-console: false`** 必须开启——GUI 闪退时无 console 看不到任何错误
2. **try-catch 全覆盖**：`handleClick()` 和 `render()` 周围捕获 `Throwable`
3. **编译-运行-报错-修复 循环**：每次改动后编译运行

### 6.2 常见错误模式识别

| 错误信息 | 根因 | 版本 |
|---------|------|------|
| `Cannot redeclare class X` | AOT 已编译所有文件，不需要 `require_once` | v1 |
| `Expression must be a global constant` | `__DIR__` 在属性默认值中 | v1 |
| `Cannot assign X to property Y of type Z` | PHP 8 类型化属性 + 反射赋值的冲突 | v1 |
| `Cannot append element to an null` | `__get` 在 AOT 中未生效 | v1 |
| `Call to undefined function any()` | `any()` 不是 PHP 函数 | v1 |
| `Unknown modifier '('` | 正则分隔符 `#` 与 CSS 颜色 `#` 冲突 | v2 |
| Elements: 2 instead of 4 | 正则字符类 `[a-z-]+` 不匹配 `:bind` | v2 |
| `error C3927: "->"` / `C3484` | 文件名点号生成无效 C++ 标识符 | v2 |
| LAYOUT array not registered in arginfo | `const` 不支持复杂嵌套数组 | v2 |
| 弹窗区域底层按钮残留 | 两个独立分层循环破坏 z-order 覆盖 | v5 |
| autoload 路径失效 / 构建找不到 lib | 硬编码绝对路径 + 搜索路径遗漏 | v5 |
| 文本无法居中对齐 | `drawText()` 缺少 `center` 分支 | v5 |
| 显示数字与 ? 按钮重叠 | `container-w` 过宽未考虑按钮空间 | v5 |

### 6.3 设计原则

1. **AOT 约束前置**：设计阶段就列出"不能用什么"
2. **极简主义**：每多一层抽象就多一个 AOT 不兼容的风险点
3. **编译期预处理**：任何需要动态处理的逻辑（模板解析、布局计算）都应在 AOT 编译前由独立工具完成
4. **显式优于隐式**：手动 `$this->dirty = true` 虽然不够优雅，但绝对可靠
5. **文件名只用 `[a-zA-Z0-9_]`**：避免点号、连字符等 AOT 不兼容字符 🆕
6. **配置数据用函数不用 const 数组**：AOT 对 const 的支持有限 🆕
7. **路径永远用相对计算**：`__DIR__` / `%~dp0`，绝不硬编码绝对路径 🆕 v5
8. **分层渲染必须单一循环**：每层内所有内容一笔画完，不得拆成多循环 🆕 v5
9. **渲染原语全覆盖**：对齐（left/center/right）等基础功能在设计期就规划完整 🆕 v5

## 七、SFC 编译器关键设计决策

基于上述踩坑经验，SFC 编译器设计遵循：

1. **编译器运行在编译前** —— 标准 PHP CLI 脚本，不经过 AOT
2. **生成的代码不能依赖 `__get/__set`** —— 显式声明所有属性，显式插入 dirty
3. **模板解析结果必须是静态数据** —— 布局坐标在编译期确定
4. **函数调用白名单** —— 生成的 PHP 代码只用已验证的 PHP 7 标准函数
5. **文件名规范** —— 输出文件名只含 `[a-zA-Z0-9_]`，不含点号 🆕
6. **配置用函数** —— 不用 `const` 存储复杂数组，用 `function getLayout(): array` 🆕
7. **颜色预计算** —— CSS `#RRGGBB` → BGR integer + 边框色自动推导 🆕

## 八、总结

| 维度 | 经验 |
|------|------|
| **最大陷阱 (v1)** | `__get/__set` 在 AOT 中不可靠，直接声明属性 + 手动标记是唯一安全方式 |
| **最大陷阱 (v2)** | 正则分隔符/字符类设计要考虑所有可能的模式内容（CSS 颜色 #、属性名 :) |
| **最大陷阱 (v5)** | 分层渲染的 z-order 取决于绘制循环结构——两个独立循环会破坏层间覆盖 |
| **最易忽略 (v2)** | 文件名会影响 AOT 生成的 C++ 符号——只用 `[a-zA-Z0-9_]` |
| **最易忽略 (v5)** | 硬编码绝对路径和构建脚本路径遗漏是迁移后最常见的失败原因 |
| **最易忽略 (v1)** | AOT 运行时 ≠ 标准 PHP 运行时，扩展不可用，PHP 8 函数可能不支持 |
| **最佳实践** | 简单 > 巧妙，显式 > 隐式，编译期预处理 > 运行时动态，单一循环 > 拆分循环（分层渲染） |
| **调试铁律** | 开 console、加 try-catch、每次改动编译验证 |

## 七、v5 M4 ~ v6 M1 迭代经验

### 7.1 AOT: static 属性触发 Notice 警告

**问题**：`protected static $queue` 声明为静态属性，在 AOT exe 运行时产生 `Notice: Accessing static property ReactiveComponent::$queue as non static`。

**解决**：将 `$queue` 和 `$componentId` 从 `protected static` 改为 `protected`（实例属性），`initShared()` 从 `static` 改为实例方法。

**教训**：AOT 对 static 属性的访问检查比 ZendPHP 更严格，避免混用 static/instance 上下文访问同一属性。

---

### 7.2 AOT: 嵌套数组类型丢失 (核心陷阱)

**问题**：函数返回嵌套数组后，提取子数组 (`$arr['key']`) 时 AOT 类型推导失败，子数组变为 `int`。运行时报错：`Only array/object/string support the item() method, type int given`

**场景**：`getLayout_app()` 返回 `['elements' => [...], 'buttons' => [...]]`，用 `$seg['elements']` 提取时类型丢失。

**解决**：避免从函数返回值多层提取嵌套数组。将完整数据内联到 `getLayout()` 中，不经过中间函数调用。

**教训**：AOT 文档专有特性章节提到 `objvar()` 可重建对象类型，但对数组类型丢失暂无等价工具。设计 AOT 兼容代码时，避免函数间传递复杂嵌套数组结构。

---

### 7.3 AOT: 变量函数调用不可靠

**问题**：`$fn = 'getLayout_app'; $fn();` 变量函数调用在 AOT 中返回值类型不正确（返回 int 而非 array）。

**解决**：改为直接硬编码函数调用，或使用 if/switch 静态分发。

---

### 7.4 RenderContext 抽象层设计

**经验**：v6 M1 引入 `RenderContext` 抽象基类 + `GdiRenderContext` 实现，将 5 个 GDI 原语（beginFrame/endFrame/fillRect/drawText/drawButton）从 BaseRenderer 中解耦。abstract class + extends 模式已验证 AOT 兼容。

---

### 7.5 分段布局的 AOT 妥协

**设计**：编译器按 group_id 生成 `getLayout_app()` / `getLayout_displayPanel()` 等分段函数，但由于 7.2 的类型丢失问题，`getLayout()` 聚合器改为直接内联全部数据而非调用分段函数。分段函数暂为死代码，待 AOT 数组类型问题解决后启用。

---

### 7.6 反模式：硬编码布局段名称列表

**问题**：`Application.php` 手动列出所有 group_id（`attachLayout('display-panel', 2)` 等），新增/删除组件需手动同步，且 camelCase/kebab-case 不一致导致闪退。

**正确做法**：SFC 编译器生成 `getLayoutSegmentNames()` 提供可用段列表，应用层动态遍历注册：
```php
$segNames = getLayoutSegmentNames();
for ($i = 0; $i < count($segNames); $i++) {
    $this->renderer->attachLayout($segNames[$i], $i + 1);
}
```

**原则**：生成代码的元信息应由生成器自身提供，消费端不应假设具体内容。

---

### 7.7 AOT 嵌套数组类型丢失根因澄清

**现象**：calculator 项目中 `callLayoutSegment()` 返回嵌套数组后访问子数组崩溃（"item() method type int"）。

**误判**：以为是 AOT 编译器对嵌套数组类型推导失败（包装函数模式触发类型链断裂）。

**实际根因**：`Application.php` 中 `attachLayout('displayPanel', ...)` 使用 camelCase，而 `callLayoutSegment()` 的 group_id 使用 kebab-case（`'display-panel'`）。名称不匹配导致分发函数走到 fallback 分支返回空数组，后续代码对空结构的处理触发异常。

**验证**：编写独立 test 套件（33 项测试），包装函数模式、(array) 转换、类方法+foreach+wrapper 全部通过。确认 AOT 编译器对嵌套数组的处理无问题。

**教训**：遇到 AOT 运行时崩溃，先排查数据流正确性（参数匹配、名称一致性），再怀疑编译器类型推导。用独立最小用例复现是最快定位手段。

### 7.8 独立测试套件的排查价值

建立了 `apps/test/` 独立 AOT 兼容性测试套件（`build.bat test --run`），33 项用例覆盖：
- 基础类型、数组、嵌套数组、包装函数模式
- 变量函数调用、变量方法调用、动态类实例化
- 类继承、抽象类、魔术方法、静态成员
- 控制流、闭包、可变参数
- 项目已遇问题专项验证

价值：当项目代码出现 AOT 崩溃时，先在 test 套件中构造最小复现用例，快速判断是编译器限制还是业务代码 bug。避免在大量生成代码中盲目排查。

### 7.9 varExportShort 短数组语法

`var_export()` 输出 `array (...)` 旧语法，在生成代码中不美观且与项目风格不一致。SFC 编译器中使用自定义 `varExportShort()` 函数：

```php
function varExportShort(array $data): string {
    $export = var_export($data, true);
    $export = preg_replace('/array\s*\(/', '[', $export);
    $export = preg_replace('/\)(,?)$/m', ']$1', $export);
    return $export;
}
```

生成的 `AppLayout_gen.php` 全部使用 `[]` 短数组语法。

### 7.10 类方法上下文中 AOT 类型推断缺陷 🆕 v6 M1

**问题**：在类方法（如 `BaseRenderer->render()`）中调用外部函数返回数组后，对该数组进行 `foreach` 遍历时，运行时错误：`Only array/object/string support the item() method, type int given`

**根因**：AOT 编译器在类方法上下文中对跨文件外部函数返回值类型推断异常。`php_calllayoutsegment()` 返回的 `php::Array` 在类方法内被错误推断为 `int`。

**现象对比**：
- test 项目（顶层函数 `php_main()`）：`foreach ($seg['elements'])` 正常
- calculator 项目（类方法 `php_baserenderer__render()`）：`seg.item()` 报 int 类型错误

**解决**：使用 `array_keys()` + for 循环替代 `foreach`：

```php
// ❌ AOT 不兼容（类方法上下文）
foreach ($this->activeLayouts as $name => $idx) {
    $seg = callLayoutSegment($name);
}

// ✅ AOT 兼容
$layoutNames = array_keys($this->activeLayouts);
$count = count($layoutNames);
for ($i = 0; $i < $count; $i++) {
    $name = strval($layoutNames[$i]);
    $seg = callLayoutSegment($name);
}
```

**注意**：此问题与 7.2（嵌套数组类型丢失）不同——这不是数组结构本身的问题，而是类方法上下文中的类型推断缺陷。同一代码模式在顶层函数中正常工作。

**教训**：AOT 类型推断存在上下文敏感性。在类方法中调用外部函数返回值后进行数组操作，应使用更显式的方式（for + 索引）而非隐式方式（foreach）。
