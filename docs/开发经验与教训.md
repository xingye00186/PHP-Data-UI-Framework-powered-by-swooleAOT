# VueCalc 开发经验与教训总结

> Swoole Compiler AOT 环境下，PHP + C++ 混合架构桌面应用开发实践

---

## 一、项目背景

VueCalc 是一个类 Vue 响应式数据驱动的 Windows 桌面计算器。技术栈：

- **PHP**：业务逻辑、响应式框架、渲染调度
- **C++ (Win32)**：窗口管理、GDI 绘制原语
- **Swoole Compiler**：PHP → C++ AOT 编译为原生 exe
- **MSVC**：最终链接为 Windows PE 可执行文件

编译产物 `vue_calc.exe` ~176KB，无需 PHP 解释器，Windows x64 原生 GUI。

## 二、踩过的坑（按时间线排序）

### 2.1 `require_once` 在顶层导致 "All execution code must be within a function"

**问题**：AOT 编译器要求所有可执行代码必须在函数内部。`require_once` 语句写在 PHP 文件顶层引发报错。

```php
// ❌ 顶层 require_once 不行
require_once __DIR__ . '/src/Calculator.php';

function main(): int { ... }
```

**解决**：移除所有顶层 `require_once`。AOT 编译器在编译阶段通过 `project.yml` 的 `sources` 配置读取所有文件，一起编译，所以不需要 `require`。

**教训**：**AOT 编译 = 编译期全量链接**。源文件声明比运行时加载更可靠。`project.yml` 的 `sources` 是唯一的文件依赖声明方式。

---

### 2.2 `__DIR__` 魔术常量不能用于属性默认值

**问题**：`__DIR__` 用于类属性的默认值时，AOT 编译器报错 "must be a global, non-magic constant"。

```php
// ❌ 属性默认值不能用 __DIR__
class Calculator extends ReactiveComponent {
    public string $template = __DIR__ . '/Calculator.vue';
}
```

**解决**：移到构造函数中赋值。

```php
// ✅ 构造函数中赋值
public function __construct(string $componentId = 'Calculator') {
    $this->template = __DIR__ . '/Calculator.vue';
    parent::__construct($componentId);
}
```

**教训**：**AOT 编译期能计算的常量有限**。`__DIR__` 虽然编译期可知，但编译器对属性默认值的约束更严格。放在函数体（构造函数）中就是安全的。

---

### 2.3 `Swoole\Table` 和 `Swoole\Atomic` 在 AOT 环境中不可用

**问题**：原设计使用 Swoole 扩展的共享内存数据结构（`Swoole\Table`、`Swoole\Atomic`）。AOT 编译的 exe 不包含 Swoole 扩展，运行时类找不到，程序启动即崩溃。

**解决**：重写为纯 PHP 实现：
- `ReactiveValue`：普通 PHP 对象替代 `Swoole\Table`
- `ChangeQueue`：数组环形缓冲替代 `Swoole\Table` + `Swoole\Atomic`

**教训**：**AOT 运行时不等于 PHP CLI 运行时**。AOT 编译产物只包含 PHP 核心运行时（`php8ts.dll`），不包含任何 PHP 扩展。使用 PHP 内置函数、标准类没有问题，但扩展类（Swoole、Redis、PDO 等）不可用。设计时必须以 **纯 PHP 标准库** 为前提。

---

### 2.4 静态属性反射问题

**问题**：`initReactiveProperties()` 使用反射遍历所有属性时，误把 `ReactiveComponent` 的静态属性（`$queue`、`$componentId`）也当作响应式属性初始化。

**解决**：添加 `$prop->isStatic()` 过滤。

**教训**：**反射操作要考虑继承链**。子类用反射遍历父类属性时，静态属性和实例属性需要区分对待。

---

### 2.5 PHP 8.x 类型化属性拒绝 `ReactiveValue` 对象

**问题**：给声明为 `string` 的属性通过反射赋值 `ReactiveValue` 对象时，PHP 8.x 强类型检查抛出 TypeError。

```php
// ❌ typed property 不能接受不匹配的值
protected string $display;  // 声明为 string
// 反射 setValue → new ReactiveValue('0') → TypeError
```

**解决**：不在子类中声明带类型的属性，改用 `__get/__set` + 内部代理数组 `$proxies`。但后来发现 AOT 不支持魔术方法，最终改为声明 `public` 属性并手动 `dirty` 标记。

**教训**：**类型系统与动态代理不相容**。PHP 8.x 的类型化属性是一把双刃剑——它提供类型安全，但也阻止了反射/代理模式的灵活性。在 AOT 约束下，`public` 无类型属性是最安全的选择。

---

### 2.6 `any($value)` —— 根本不是 PHP 函数

**问题**：`ChangeQueue::push()` 中使用了 `any($value)`。这**不是** PHP 标准函数，AOT 编译后的运行时会直接 fatal error。

```php
// ❌ any() 不存在于 PHP
'value' => json_encode(any($value))
```

这导致每次点击按钮 → 修改属性 → `__set` → `push` → `any()` → **崩溃**。

**解决**：移除 `any()`，直接存储值。同时移除 `json_encode`/`json_decode`（AOT 中可能不可用）。

**教训**：**代码生成/LLM 输出必须校验**。`any()` 可能是从 TypeScript 或其他语言混入的。在 AOT 环境中，每一个函数调用都必须确认是 PHP 内置或自定义函数。

---

### 2.7 `str_contains()` 是 PHP 8.0 函数

**问题**：`str_contains($this->display, '.')` 在 PHP 7.x 或精简的 AOT 运行时中可能不存在。

**解决**：替换为 `strpos($this->display, '.') !== false`（PHP 4 兼容）。

**教训**：**目标 PHP 版本要明确**。AOT 编译的 PHP 运行时版本可能低于代码编写的版本。使用 PHP 7 级别的 API（`strpos`、`substr`、`strlen`）而不是 PHP 8 新增函数（`str_contains`、`str_starts_with`、`str_ends_with`）更安全。

---

### 2.8 `__get/__set` 魔术方法在 AOT 中不被调用（最大坑）

**问题**：这是最核心、最难定位的 bug。

设计思路是：Calculator 不声明任何属性，通过 `reactiveProperties()` 动态声明，所有读写由 `ReactiveComponent.__get/__set` 统一拦截。在标准 PHP 中完美运行。

但在 AOT 编译后：

```php
// 代码
$this->display .= $digit;

// 标准 PHP：
//   $temp = $this->display;    // → __get('display')
//   $this->display = $temp . $digit;  // → __set('display', ...)

// AOT 编译后：
//   __get 不被调用 → $temp = null
//   null .= '5' → TypeError: Cannot append element to an `null`
```

**解决**：彻底放弃魔术方法，改用：
1. Calculator 类声明 6 个 `public` 属性
2. 每个修改状态的方法末尾手动 `$this->dirty = true`
3. `ReactiveComponent` 简化为仅保留 `$dirty` 属性和共享基础设施

**教训**：
- **AOT 编译器的行为 ≠ 标准 PHP 的行为**。魔术方法、反射、动态特性是 "AOT 高危区域"
- **`__get/__set` 不要指望**。在 AOT 中，声明属性直接编译为 C++ 成员变量，未声明的属性编译器可能不调用魔术方法
- **"优雅"要妥协于"可靠"**。手动 `dirty` 标记虽然多了几行代码，但在 AOT 约束下这是唯一可靠的方式
- **出问题时先怀疑基础假设**。这个 bug 排查了很久，因为一直认为 `__get/__set` 是 PHP 基本特性，应该没问题

---

### 2.9 静态属性类型声明

**问题**：`protected static ChangeQueue $queue;` 在 PHP 8.x 中不设默认值是"未初始化"状态，AOT 编译器也可能处理不好。

**解决**：改为 `protected static ?ChangeQueue $queue = null;`，在使用前加 `!== null` 检查。

**教训**：**静态属性给默认值**——避免 PHP 8 类型化静态属性的"未初始化"语义在 AOT 中引发问题。

## 三、架构演进的教训

### 3.1 初始架构（失败）

```
Calculator.reactiveProperties()  →  ReactiveComponent.__get/__set  →  ReactiveValue 代理
    声明属性名+默认值                   拦截所有读写                       值+版本号
                                  + ChangeQueue 变更队列
```

**失败原因**：
1. `__get/__set` 在 AOT 中不可靠 —— 致命
2. `ReactiveValue` 通过反射填充到 typed property 触发 TypeError
3. `ChangeQueue` 依赖 `any()` / `json_encode` —— 不存在或不可用

### 3.2 最终架构（可行）

```
Calculator.public 属性声明  →  方法内部手动 $this->dirty = true  →  主循环检测 dirty → render()
  6个 string/bool 属性           每个修改状态的方法末尾                  CalcApp::run() 每帧检查
```

**核心变化**：

| 元素 | 初始方案 | 最终方案 |
|------|---------|---------|
| 属性声明 | 动态（reactiveProperties） | 显式 public 声明 |
| 读写拦截 | `__get/__set` 自动 | 直接访问，不拦截 |
| 脏标记 | `__set` 中自动 | 方法末尾手动 |
| 数据存储 | `ReactiveValue` 代理对象 | 属性本身 |
| 变更通知 | `ChangeQueue` push/pop | 未使用（保留供扩展） |

### 3.3 架构教训

1. **由简到繁，而非相反**。初始设计过度抽象——代理、反射、队列，每层抽象在 AOT 下都是一个风险点。最终极简架构反而稳定。
2. **AOT 优先**：设计阶段就应明确"AOT 能做什么，不能做什么"，而不是写好再移植
3. **手动比自动更可控**：在受限环境下，显式代码 > 隐式惯例。每个 `$this->dirty = true` 是多余的，但它保证了行为确定性
4. **保留扩展接口**：`ChangeQueue` 和 `ReactiveValue` 保留在代码中，供将来增量渲染时使用

## 四、事件循环的设计经验

### 4.1 最终形态

```php
while ($running) {
    // 内层：批量处理所有 Windows 消息
    while (true) {
        $msg = vue_peek_message();   // C++ PeekMessage 封装，1次1条
        if (count($msg) == 0) break; // 队列空 → 退出内层
        if (WM_LBUTTONDOWN) { handleClick(); }
        if (WM_QUIT) { $running = false; }
    }
    // 外层：消息清空后检查 dirty → 渲染
    if ($this->calc->dirty) {
        $this->renderer->render();
        $this->calc->dirty = false;
    }
    usleep(16000);
}
```

### 4.2 设计要点

1. **C++ `vue_peek_message()`** 每次调用返回 **1 条**消息（4 元素数组 `[hwnd, msg, wParam, lParam]`），不是多条
2. 内层 `while(true)` 循环一次性清空消息队列，外层只渲染一次——**批量更新**模式
3. `usleep(16000)` ≈ 60 FPS 节奏，避免忙等消耗 CPU
4. 不要在每条消息后都渲染——消息处理极快（几个属性赋值），批量渲染即可

## 五、C++/PHP 互操作经验

### 5.1 Stub 模式

```php
// php-src/vue_calc.stub.php — PHP 侧声明
function vue_window_create(string $title, int $width, int $height): int {}
```

```cpp
// cpp-src/vue_calc.cc — C++ 侧实现
Int php_vue_window_create(String title, Int width, Int height) { ... }
```

**规则**：PHP 函数名 `vue_xxx` → C++ 函数名 `php_vue_xxx`。AOT 编译器自动识别并链接。

### 5.2 类型映射

| PHP Stub 类型 | C++ phpx 类型 | 说明 |
|-------------|-------------|------|
| `int` | `Int` | 整数句柄（窗口/GDI） |
| `string` | `String` | `.data()` 取 `const char*` |
| `bool` | `Bool` | 布尔值 |
| `array` | `Array` | 用 `.append()` 构建 |
| `void` | `void` | 无返回值 |

### 5.3 GDI 双缓冲

```cpp
// begin_paint: 创建内存 DC
HDC memDC = CreateCompatibleDC(hdc);
HBITMAP bitmap = CreateCompatibleBitmap(hdc, w, h);
SelectObject(memDC, bitmap);

// ... 所有绘制在 memDC 上进行 ...

// end_paint: 一次性贴到屏幕
BitBlt(hdc, 0, 0, w, h, memDC, 0, 0, SRCCOPY);
DeleteDC(memDC);
DeleteObject(bitmap);
```

不这样做会产生闪烁——因为每个 GDI 绘制调用都可能触发部分屏幕更新。

## 六、开发流程教训

### 6.1 调试方法

1. **`no-console: false`** 必须开启。GUI 应用闪退时，没有控制台就看不到任何错误信息
2. **try-catch 全覆盖**：在 `handleClick()` 和 `render()` 周围捕获 `Throwable`，打印到控制台
3. **编译-运行-报错-修复 循环**：每次改动后编译（~10秒），运行测试，从错误信息反推问题

### 6.2 常见错误模式识别

| 错误信息 | 根因 | 
|---------|------|
| `Cannot redeclare class X` | AOT 已编译所有文件，不再需要 `require_once` |
| `Expression must be a global constant` | 属性默认值用了 `__DIR__` 等魔术常量 |
| `Cannot assign X to property Y of type Z` | PHP 8 类型化属性 + 反射赋值的冲突 |
| `Cannot append element to an null` | `__get` 在 AOT 中未生效，属性读取返回 null |
| `Call to undefined function any()` | `any()` 不是 PHP 函数 |

### 6.3 设计原则

1. **AOT 约束前置**：设计阶段就列出"不能用什么"——扩展到 SFC 编译器的设计
2. **极简主义**：每多一层抽象就多一个 AOT 不兼容的风险点
3. **编译期预处理**：任何需要动态处理的逻辑（模板解析、布局计算），都应在 AOT 编译前由独立工具完成
4. **显式优于隐式**：手动 `$this->dirty = true` 虽然不够优雅，但绝对可靠

## 七、SFC 编译器展望：关键教训导出的设计原则

基于上述踩坑经验，面向 AOT 的 .vue SFC 编译器必须遵循：

1. **编译器运行在编译前** —— 不是运行时组件，是 AOT 的预处理器
2. **生成的代码不能依赖 `__get/__set`** —— 必须显式声明所有属性，显式插入 dirty 标记
3. **模板解析结果必须是静态数据** —— 布局坐标在编译期确定，生成 C++ 常量数组
4. **函数调用白名单** —— SFC 编译器生成的 PHP 代码只能用已验证的 PHP 7 标准函数
5. **`.vue` 的 `<script>` 只允许纯 PHP 逻辑** —— 不涉及 UI 操作，业务逻辑与渲染分离
6. **增量渲染需要节点级 dirty** —— 编译器需分析每个 `v-model` 的数据依赖，生成精准的脏标记

## 八、总结

| 维度 | 经验 |
|------|------|
| **最大陷阱** | `__get/__set` 在 AOT 中不可靠，直接声明属性 + 手动标记是唯一安全方式 |
| **最易忽略** | AOT 运行时 ≠ 标准 PHP 运行时，扩展不可用，PHP 8 函数可能不支持 |
| **最佳实践** | 简单 > 巧妙，显式 > 隐式，编译期预处理 > 运行时动态 |
| **调试铁律** | 开 console、加 try-catch、每次改动编译验证 |
| **架构方向** | SFC 编译器必须作为 AOT 预处理器运行，生成的代码遵循严格的白名单约束 |
