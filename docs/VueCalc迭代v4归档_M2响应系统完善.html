一、迭代背景

v3 设计规划文档（VueCalc技术规划文档_v3.html §5.4）将 M2 定义为「响应系统完善」里程碑 — 核心目标是消除运行时三大硬编码技术债务，让模板与组件之间形成真正的声明式闭环。

三大运行时债务（v3 文档 §3.3 D4-D6）：
D4 — 手动 dirty 标记：每个修改状态的方法都必须手写 $this->dirty = true;（7处），容易遗漏
D5 — 手动 getBindValue：渲染器的属性查找使用手写 if/else 链，新增绑定时必须同步修改
D6 — 手动 dispatchClick：按钮事件路由使用手写 if/else 链，新增按钮时必须同步修改

M2 的 5 项子目标：

  1. M2.1 自动 Dirty 标记：编译器分析 script 块，自动识别修改 reactive 属性的方法并注入 $this->dirty = true;
  2. M2.2 自动 getBindValue 生成：编译器遍历 template AST，收集所有 :bind 键 → 生成 if/else 属性映射方法
  3. M2.3 自动 dispatchClick 生成：编译器收集所有 @click handler → 生成 if/elseif 事件路由方法
  4. M2.4 v-model 双向绑定：新增 v-model="prop" 语法（与 :bind 互斥），隐含单向绑定 + 自动 dirty
  5. M2.5 v-if 条件渲染：新增 v-if="condition" 语法，支持 truthy / == / != 三种条件判断

四、M2.1 自动 Dirty 标记 — 技术详解

4.1 状态机设计

ScriptAnalyzer 采用三状态状态机解析 PHP 方法：

状态转换:
  'outside'  → 遇到 'function' 关键字 → 'header' (方法签名)
  'header'   → 遇到 '{' → 'body' (方法体，depth=1)
  'body'     → 遇到 '{' → depth++ | 遇到 '}' → depth-- | depth==0 → 'outside'

核心判断 — 方法是否修改属性:
  methodModifiesProperties($body):
    return preg_match('/\$this->(\w+)\s*[.=]/', $body) === 1;

注入策略:
  processMethodBody($methodName, $body):
    1. 在 return ...; 之前注入 $this->dirty = true;
    2. 在方法末尾（闭合 } 之前）注入 $this->dirty = true;
    3. 跳过 __construct (构造器不触发布局刷新)
    4. 跳过纯调度方法 (如 handleButton — 只调用其他方法，不直接修改属性)

五、M2.2 / M2.3 自动胶水代码 — 技术详解

5.1 getBindValue 自动生成

编译器在 lowerToLayout() 遍历 AST 时收集所有 TextNode 的 :bind 属性值，去重后输出为 bindKeys 数组。sfc-compiler 根据 bindKeys 生成 if/else 映射链：

// 编译器收集阶段 (template-parser.php lowerToLayout)
foreach ($app->children as $child) {
    if ($child instanceof TextNode && $child->bind !== '') {
        $bindKeys[$child->bind] = true;  // → ['expression', 'display']
    }
}

// 编译器生成阶段 (sfc-compiler.php)
foreach ($bindKeys as $key) {
    $getBindValueBody .= "        if (\$bindKey === '$key') {\n";
    $getBindValueBody .= "            return \$this->$key;\n";
    $getBindValueBody .= "        }\n";
}

// 生成结果 (Calculator.gen.php)
public function getBindValue(string $bindKey): string
{
    if ($bindKey === 'expression') return $this->expression;
    if ($bindKey === 'display') return $this->display;
    return '';
}

// main.php 简化为委托 (CalcRenderer)
private function getBindValue(string $bindKey): string
{
    return $this->component->getBindValue($bindKey);
}

5.2 dispatchClick 自动生成

编译器在 lowerToLayout() 遍历 GridNode 的按钮时收集所有 @click handler，记录是否需要传参（hasArg）：

// 编译器收集阶段
if (!isset($handlerMap[$btn->handler])) {
    $handlerMap[$btn->handler] = ($btn->arg !== null);
}

// handlerMap 结果: {'reset'=>false, 'backspace'=>false, 'handleButton'=>true, 'calculate'=>false}

// 生成结果 (Calculator.gen.php)
public function dispatchClick(array $btn): void
{
    $handler = $btn['handler'];
    if ($handler === 'reset') {
        $this->reset();
    }
    elseif ($handler === 'backspace') {
        $this->backspace();
    }
    elseif ($handler === 'handleButton') {
        $this->handleButton($btn['arg']);
    }
    elseif ($handler === 'calculate') {
        $this->calculate();
    }
}

// main.php 简化为委托 (CalcApp)
private function dispatchClick(array $btn): void
{
    $this->calc->dispatchClick($btn);
}
设计决策：if/elseif 而非 switch。因为 AOT 编译器需要显式字符串比较，switch 对字符串参数的支持在 AOT 环境中不可靠。

六、M2.4 / M2.5 声明式能力 — 技术详解

6.1 v-model 双向绑定

v-model 和 :bind 互斥，两者不能同时出现在同一元素上。v-model 隐含 :bind 的单向渲染语义，双向绑定中的「写回」由 M2.1 的自动 dirty 机制间接保证。

// AST 扩展 (ast-nodes.php)
class TextNode extends TemplateNode {
    public string $vModel = '';  // v4 M2.4
}

// 解析逻辑 (template-parser.php parseText)
$vModel = $attrs['v-model'] ?? '';
if ($vModel !== '' && $bind !== '') {
    $this->error("cannot have both v-model and :bind", $tok->line);
}
if ($vModel !== '') {
    $bind = $vModel;  // v-model 隐含 :bind
}

// 模板使用
<text x="10" y="10" v-model="expression" v-if="expression" ... />

6.2 v-if 条件渲染

v-if 适用于任意元素（rect / text / grid / btn），条件表达式支持三种形式：

语法编译为 condition运行时求值
v-if="propName"['prop'=>'propName', 'op'=>'truthy']$this->propName ? true : false
v-if="propName == 'val'"['prop'=>'propName', 'op'=>'==', 'value'=>'val']$this->propName === 'val'
v-if="propName != 'val'"['prop'=>'propName', 'op'=>'!=', 'value'=>'val']$this->propName !== 'val'
// AST 扩展 (ast-nodes.php)
abstract class TemplateNode {
    public string $vIf = '';  // v4 M2.5 所有元素共用
}

// 条件解析 (template-parser.php)
private function parseVIfCondition(string $vIf): array {
    if (preg_match("/^(\w+)\s*(==|!=)\s*'([^']*)'$/", $vIf, $m))
        return ['prop' => $m[1], 'op' => $m[2], 'value' => $m[3]];
    if (preg_match('/^(\w+)$/', $vIf))
        return ['prop' => $vIf, 'op' => 'truthy'];
    return ['prop' => $vIf, 'op' => 'truthy'];  // fallback
}

// 编译器自动生成 evalCondition (Calculator.gen.php)
public function evalCondition(array $cond): bool
{
    $prop = $cond['prop'];
    $op = $cond['op'];
    if ($op === 'truthy') {
        if ($prop === 'expression') return $this->expression ? true : false;
        return false;
    }
    if ($op === '==') { ... }
    if ($op === '!=') { ... }
    return false;
}

// 运行时渲染跳过 (main.php CalcRenderer::render)
foreach ($elements as $el) {
    if (isset($el['condition']) && !$this->component->evalCondition($el['condition']))
        continue;  // v-if 不满足
    // ... 正常渲染
}
设计约束：AOT 环境不支持运行时表达式求值和变量属性访问（$obj->$prop），因此条件表达式必须在编译期解析为结构化数据，运行时通过显式 if/else 链求值。evalCondition 在 Calculator.gen.php 中生成（而非 main.php），确保每个 prop 的访问都是编译期确定的。

#里程碑问题解决

5 M2.3 elseif 缩进多一个空格（prefix ' elseif' 带了前置空格） 将 prefix 改为 'elseif'（不含空格），由格式字符串中的空格统一控制间距
