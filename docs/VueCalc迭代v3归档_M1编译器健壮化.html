一、迭代背景

v3 设计规划文档（VueCalc技术规划文档_v3.html §5.3）将 M1 定义为最高优先级里程碑——「SFC 编译器健壮化」。其核心问题：

技术债务 D1（严重度：高）：tools/sfc-compiler.php:111-227 中模板解析完全基于正则表达式，标签名硬编码在 app|rect|text|grid|btn 中，无 AST 中间表示，无行号跟踪，无错误恢复。

M1 的 5 项目标：

  1. 解析器升级：纯正则 → 递归下降 parser，输出 AST 中间表示
  2. AOT 兼容性校验：编译器输出前自动检测 AOT 约束违规
  3. 错误报告：解析失败时输出行号和描述性错误信息
  4. CSS 映射表正式化：建立结构化 [css-prop → gdi-param] 映射
  5. 单元测试覆盖：为 sfc-compiler 各步骤增加测试用例

四、关键技术细节

4.1 递归下降 Parser 设计

TemplateParser 采用经典三阶段架构：

Phase 1: Tokenize

  • 逐字符扫描模板字符串，跟踪行号
  • 识别 6 种 token 类型: TOK_TAG_OPEN, TOK_TAG_CLOSE, TOK_TAG_SELF, TOK_TEXT, TOK_COMMENT, TOK_EOF
  • 注释 (<!-- ... -->) 被正确跳过，不影响解析

Phase 2: Parse

parseDocument()          // 入口: 期望 <app>
  → parseApp()           // 解析 <app ...>...</app>
    → parseElement()     // 分发: rect | text | grid | unknown
      → parseRect()      // <rect x=... class=... />
      → parseText()      // <text :bind=... class=... />
      → parseGrid()      // <grid ...>
        → parseBtn()     //   <btn row=... col=... @click=... />
      → UnknownNode      // 未知标签: 报告 → 继续解析

Phase 3: Lower to Layout

lowerToLayout(AppNode, classStyles) 遍历 AST，应用 CSS 样式，预计算 Grid 按钮像素坐标，输出 ['elements'=>[...], 'buttons'=>[...]] 数组——格式与原版完全兼容。

4.2 错误报告能力

新增 8 种编译时错误检测，全部带行号：

错误触发条件严重度
Empty template模板中无 <app> 根元素Fatal
Expected <app>根标签不是 <app>Fatal
Unclosed <app>缺少 </app>Fatal
Unclosed <grid>缺少 </grid>Fatal
<btn> outside <grid><btn> 直接在 <app> 下Error
Unknown element不支持的标签名Warning
Missing class<rect>/<text> 无 class 属性Warning
No :bind<text> 无 :bind 属性（永不渲染）Warning

4.3 AOT 校验规则

// 5 条校验规则 (aot-validator.php)
Rule 1: 文件名 stem 中 ≤1 个点 (Calculator.gen.php ✓, Foo.Bar.gen.php ✗)
Rule 2: 无 const 嵌套数组          (const LAYOUT = [...] ✗)
Rule 3: 无变量属性访问             ($obj->$var ✗)
Rule 4: 无变量方法调用             ($obj->$method() ✗)
Rule 5: 无 PHP8 独有函数           (str_contains → 建议 strpos)
注意：校验规则本身也用 PHP7 兼容写法（substr($s, -4) !== '.php' 而非 str_ends_with()），确保 AOT validator 可在各 PHP 版本运行。

4.4 CSS 映射表

从散落的 if/else 升级为正式 PROPERTY_MAP 常量（关联数组），扩展新属性只需加一行：

const PROPERTY_MAP = [
    'background'    => ['key'=>'bg',           'parser'=>'CssMappings::parseHexColor',   'default'=>0],
    'color'         => ['key'=>'fg',           'parser'=>'CssMappings::parseHexColor',   'default'=>0xFFFFFF],
    'font-size'     => ['key'=>'fontSize',     'parser'=>'CssMappings::parsePixels',     'default'=>16],
    'font-weight'   => ['key'=>'bold',         'parser'=>'CssMappings::parseFontWeight', 'default'=>0],
    // M1 新增:
    'border-radius' => ['key'=>'borderRadius', 'parser'=>'CssMappings::parsePixels',     'default'=>0],
    'padding'       => ['key'=>'padding',      'parser'=>'CssMappings::parsePixels',     'default'=>0],
    'margin'        => ['key'=>'margin',       'parser'=>'CssMappings::parsePixels',     'default'=>0],
    'text-align'    => ['key'=>'textAlign',    'parser'=>'CssMappings::parseTextAlign',  'default'=>'left'],
];

同时新增 hexToBgr() 对 #RGB 简写格式的支持（D10 技术债务修复）。

七、实施中遇到的问题与解决

#问题解决
1 ParseError 与 PHP 7.x 内置类冲突 重命名为 TemplateParseError
2 str_ends_with() 在 aot-validator 中使用 (反讽: AOT校验器自己用了PHP8函数) 改为 substr($basename, -4) !== '.php'
3 文件名校验规则太严格 — Calculator.gen.php (1个stem点) 被误拒绝 改为检查 stem 中的点数 ≤1（允许 Foo.gen.php，拒绝 Foo.Bar.gen.php）
4 Top-level code warning 误报 — const WINDOW_WIDTH = ... 是合法顶层声明 移除 Rule 6（过度敏感），const/class/function 顶层声明在生成文件中合法
5 Parser EOF token 在 parseApp 后被误报为 "Unexpected content after </app>" 添加 TOK_EOF 豁免: 仅在 token 非 EOF 且非空白时报警
6 AOT 编译器拾取 src/compiler/ — 导致 require_once 顶层语句 AOT 编译失败 将编译器模块从 src/compiler/ 移至 tools/compiler/（不在 project.yml sources 中）
教训：AOT 编译器的 sources: ["./src"] 配置会递归包含子目录。任何仅用于标准 PHP CLI 的工具代码必须放在 src/ 目录树之外。已将此约束补充到 AOT 校验规则知识库中。
