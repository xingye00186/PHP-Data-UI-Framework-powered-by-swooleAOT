一、v1 vs v2：路线图阶段执行评估

以下是 v2 实现相对于 v1 8 阶段路线图的完成度评估。注意 v1 写于 SFC 编译器构思阶段， v2 是首次 SFC 管道实战，很多假设得到验证，也暴露出新问题。

P1

Phase 1 · Template Parser · 已实现 (60%)

v1 目标: 手写 PHP Template Parser —— 解析 .vue 的 <template> 块，输出 AST

已实现

  • ✅ 正则提取 <template> / <script lang="php"> / <style> 三个块
  • ✅ 支持 5 种自定义标签：<app> <rect> <text :bind> <grid> <btn @click>
  • ✅ 属性名支持 :bind 和 @click 等 Vue 风格指令

仍未解决

  • ❌ 无 AST 输出：当前为直接正则匹配 → 数组，没有中间语法树表示。v1 目标「输出 AST」未达成
  • ❌ 无通用解析器：标签名硬编码在正则里（app|rect|text|grid|btn），不支持自定义标签扩展
  • ❌ 正则脆弱性：CSS #RRGGBB 与正则分隔符冲突、属性名字符类不完整，已在 v2 修复但表明正则方案不可靠
  • ❌ 无错误报告：解析失败时静默退出，没有行号/位置信息
v3 建议：在正则方案可用的前提下暂时维持现状。当需要支持更多标签类型或嵌套结构时，必须升级为递归下降 parser 并输出 AST。
P2

Phase 2 · Style→GDI Mapper · 已实现 (50%)

v1 目标: CSS 属性子集到 GDI 常量的字典

已实现

  • ✅ hexToBgr()：CSS #RRGGBB → Win32 COLORREF BGR 整数
  • ✅ borderColor()：自动推导较亮的边框色（每个通道 +20）
  • ✅ CSS class 名称到颜色属性的键值对映射

仍未解决

  • ❌ 仅映射颜色：font-size / font-weight 虽然在 CSS 中定义但在代码生成时硬编码处理，不是通用映射表
  • ❌ 无 opacity/alpha 支持：GDI 不支持 alpha 通道（v1 已指出）
  • ❌ 无 border-radius 映射：RoundRect API 存在但未使用
  • ❌ 无 spacing/padding 支持：当前全部是精确坐标，无 CSS Box Model
v3 建议：建立正式的 CSS→GDI 属性映射表（关联数组），而非分散在解析器中的 if/else。为 Direct2D 升级预留接口。
P3

Phase 3 · Layout Engine · 已实现 (55%)

v1 目标: 基于 AST + 映射表 → 生成坐标布局数据

已实现

  • ✅ Grid 布局：cols / rows / cell-w / cell-h / margin → 预计算像素坐标
  • ✅ 绝对定位矩形和文本元素
  • ✅ 所有坐标在编译期计算，运行时零开销
  • ✅ 这正是 v1 提出的「方案3：编译期计算好布局」—— 验证了可行性

仍未解决

  • ❌ 仅 Grid 布局：无 flex、无流式布局、无自动换行
  • ❌ 无 CSS Box Model：没有 margin/border/padding/content 四层盒子概念
  • ❌ 无响应式：窗口大小固定（328×420），不支持 resize
v2 验证成果：v1 中提出的三种布局方案，v2 实现了方案3（编译期预计算）并证明完全可行。这是 v2 最重要的架构验证之一。
P4

Phase 4 · Code Generator · 已实现 (65%)

v1 目标: AST + 布局数据 → 生成 PHP 子类（属性+方法） + 布局常量

已实现

  • ✅ Calculator.gen.php：ReactiveComponent 子类 + script 块内容 + 自动构造函数
  • ✅ CalculatorLayout_gen.php：getLayout(): array 函数 + 窗口常量
  • ✅ AOT 兼容输出：函数返回替代 const 数组、下划线替代文件名点号

仍未解决

  • ❌ 未自动插入 dirty 标记：v1 设想「方法末尾自动插入 $this->dirty = true」，当前仍由开发者手写
  • ❌ 模板和类分离：布局生成为独立文件（_gen.php + Layout_gen.php），而非像 v1 设想生成 C++ 头文件
  • ❌ 无属性自动声明：script 块中的属性原样复制，编译器未解析和增强
P5

Phase 5 · v-model 指令解析 · 部分实现 (25%)

v1 目标: 自动生成属性声明、getter/setter、dirty 标记代码

已实现

  • ✅ :bind="prop" 单向绑定：通过 getBindValue() 显式 if/else 映射
  • ✅ @click="handler" / @click="handler('arg')" 事件绑定

仍未解决

  • ❌ 无 v-model 双向绑定：没有在模板中声明「此文本既显示又修改」的能力
  • ❌ 无 v-if / v-for：条件和循环指令完全缺失
  • ❌ 无 getter/setter 生成：dirty 标记仍需手动
  • ❌ getBindValue() 硬编码：每个新属性需要手动添加 if 分支
v3 建议：单向 :bind 可维持现有方案（显式 if/else 是 AOT 最可靠模式）。v-model 双向绑定是实现重点——需要编译器自动生成 setter 中的 $this->dirty = true。
P6

Phase 6 · 组件树支持 · 未开始 (0%)

v1 目标: 嵌套组件编译展开

当前状态

  • 单组件应用：一个 App 包含一个 Calculator 组件 —— 无嵌套
  • 组件关系在 main.php 中硬编码创建，而非从模板编译

需要实现

  • 子组件标签解析（如 <my-component />）
  • 编译期组件注册表（组件名 → 类名映射）
  • 子组件在构造函数中自动实例化（AOT 要求编译期确定类型）
  • props 传递机制（父组件数据 → 子组件）
复杂度预警：组件树支持将显著增加编译器复杂度。v1 中提出的「编译期展开子组件为成员变量」方案在 AOT 约束下是正确的方向。
P7

Phase 7 · 增量渲染引擎 · 未开始 (0%)

v1 目标: 节点级 dirty 标记，区域重绘

当前状态

  • 全局 dirty 标志 → 全量 GDI 重绘
  • 对计算器（4 个元素 + 18 个按钮，328×420）性能足够
  • 对复杂 UI 不可扩展

需要实现

  • 节点级 dirty 标记：每个布局元素独立标记是否需重绘
  • 依赖分析：v-model 变更 → 自动标记受影响节点
  • 区域裁剪：仅重绘 dirty 节点的 bounding box
  • 脏区合并：相邻 dirty 区域合并为一个大矩形避免过度绘制
优先级：此阶段仅在 UI 复杂度显著提升后才需要。当前计算器场景中，全量 GDI 重绘 22 个元素无明显性能问题。
P8

Phase 8 · 渲染层升级 GDI→Direct2D · 未开始 (0%)

v1 目标: 支持更丰富的视觉效果

升级收益

  • Alpha 通道 / 半透明
  • 抗锯齿文本和形状
  • 硬件加速（GPU 渲染）
  • 更丰富的 CSS 映射（box-shadow / gradient / opacity）

升级代价

  • 增加 d2d1.dll / dwrite.dll 依赖
  • 需要重写 cpp-src 中的渲染代码
  • Direct2D API 比 GDI 复杂得多
  • exe 体积可能增加
建议：此阶段应推迟到有明确的视觉效果需求时再启动。GDI 对当前计算器完全够用。若需要 Web 级视觉效果，考虑 Skia 而非 Direct2D（更成熟的跨平台方案）。

4.6 v2 实践发现的新问题

以下 6 个问题在 v1 中未预见，在 v2 实践中暴露：

新问题 1：正则解析的不可靠性

v2 SFC 编译器完全基于正则表达式解析，导致两个实际 bug（CSS 颜色分隔符冲突、属性名不匹配）。正则方案对简单模板勉强可用，但缺乏错误恢复、无位置信息、嵌套受限。

改进方向：在需要支持嵌套标签、条件渲染、循环等高级特性时，升级为手写递归下降 parser。短期可维持正则 + 更多单元测试。

新问题 2：SFC 编译器无校验步骤

sfc-compiler.php 直接输出 .gen.php，不检验输出是否违反 AOT 约束。v2 中是通过「AOT 编译失败 → 回溯排查」的方式发现文件名点号和 const 数组问题的。

改进方向：在 sfc-compiler.php 输出前增加 AOT 兼容性校验步骤，自动检测：文件名点号、const 复杂数组、变量属性访问等已知约束。

新问题 3：模板语法非标准化

v2 的自定义标签（<rect>、<btn>、<grid>）是项目专用的。未来若要用同一个编译器处理不同类型的应用（计算器 vs 其他），需要某种方式来声明自定义标签映射。

改进方向：引入标签注册机制（配置文件或 .vue 中的注释声明），使 sfc-compiler 可配置化。

新问题 4：双文件输出架构问题

v2 生成两个文件（Calculator.gen.php + CalculatorLayout_gen.php）而非 v1 设想的「PHP 类 + C++ 头文件」。这导致布局数据通过 PHP 函数返回（运行时），而非编译进 C++ 结构体（零开销）。

改进方向：评估是否需要生成 C++ 头文件。当前 PHP 函数方式在性能上对简单 UI 无影响，但 foreach(getLayout()['buttons']) 的每次渲染遍历有微小的性能开销。

新问题 5：构建管道的环境脆弱性

v2 构建依赖：MSVC（vcvarsall.bat 初始化）、MSYS2 bash（cmd //c 而非 cmd /c）、PHP CLI 在 PATH 中。更换开发机器时需要重新配置。

改进方向：在 project.yml 中增加 prebuild hook 自动运行 sfc-compiler，或将 sfc-compiler 集成到 AOT 编译器的预处理步骤中。

新问题 6：AOT 调试困难

当 AOT 编译失败时，错误信息通常是 C++ 编译错误（如 error C3927），难以关联回原始 PHP 代码。v2 中通过经验排查（猜测是文件名点号导致无效 C++ 函数名）解决了问题。

改进方向：建立 PHP→C++ 错误映射知识库，或将常见错误模式记录在文档中（已部分在《构建编译流程参考.md》中整理）。

    Int hdc = args[0].toInt();      // arg[0]: int $hdc

    Int w   = args[3].toInt();

5.3.4 AOT 校验器设计

// src/compiler/aot-validator.php
function validateAotCompatibility(string $code, string $filename): array {
    $warnings = [];

    // 规则1: 文件名不含点号（除了 .php 后缀）
    if (substr_count(basename($filename), '.') > 1) {
        $warnings[] = "AOT: 文件名含多余点号 '$filename'，将生成无效 C++ 函数名";
    }

    // 规则2: 无 const 复杂嵌套数组
    if (preg_match('/const\s+\w+\s*=\s*\[/s', $code)) {
        $warnings[] = "AOT: const 数组可能不被支持，改用 function 返回";
    }

    // 规则3: 无变量属性访问 $obj->$var
    if (preg_match('/\$\w+->\$\w+/', $code)) {
        $warnings[] = "AOT: 变量属性访问 (\$obj->\$var) 不被支持";
    }

    // 规则4: 无变量方法调用 $obj->$method()
    if (preg_match('/\$\w+->\$\w+\s*\(/', $code)) {
        $warnings[] = "AOT: 变量方法调用 (\$obj->\$method()) 不被支持";
    }

    return $warnings;
}

            $msgType = $msg[1] ?? 0;         // msg[1] = UINT message
            if ($msgType == WM_LBUTTONDOWN) {
                // 解析鼠标坐标 (lParam 封包)
                $lParam = $msg[3] ?? 0;      // msg[3] = LPARAM
                $mx = $lParam & 0xFFFF;      // LOWORD
                $my = ($lParam >> 16) & 0xFFFF; // HIWORD
                $this->handleClick($mx, $my);
            }
            if ($msgType == WM_QUIT) {
                $running = false;
                break;
            }
        }

        // ===== 外循环: 后备退出检查 =====
        if (vue_quit_requested()) $running = false;

        // ===== 脏标记驱动重绘 =====
        if ($this->calc->dirty) {
            $this->renderer->render();       // 仅在需要时重绘
            $this->calc->dirty = false;      // 清除脏标记
        }

        usleep(16000); // ~60 FPS 帧率控制
    }
}
微妙点：vue_quit_requested() 不能替代 WM_QUIT 检查——两者互补。WM_QUIT 由窗口关闭消息触发（消息队列内），vue_quit_requested() 读取 C++ 静态变量 g_quitRequested（窗口销毁时设置）。外层 if 是后备机制，防止 WM_QUIT 因某种原因未被投递。

private function dispatchClick(array $btn): void {

8.4 运行时性能剖析

8.4.1 帧率控制

usleep(16000);  // 16ms → 理论最高 ~62.5 FPS

实际帧率受以下因素影响：

  • 空闲帧（无用户操作）：约 60 FPS，每帧仅检测 vue_peek_message 和 vue_quit_requested
  • 脏帧（状态变更后）：约 55-60 FPS，增加 42 次 PHP→C++ GDI 调用
  • 连续输入帧（快速点击）：每帧需额外执行 handleClick → dispatchClick → 业务逻辑

8.4.2 每帧操作开销估算

阶段操作PHP↔C++ 往返估算开销
消息检测（空闲）vue_peek_message() + vue_quit_requested()1-2 次~10µs
消息处理（有点击）命中测试 (PHP端) + 业务逻辑 (PHP端)0 次~1µs
脏帧渲染42 次 GDI 调用 + 双缓冲 BitBlt42 次~200-500µs
帧间等待usleep(16000)—16,000µs

关键结论：GDI 渲染开销 (~500µs) 远小于帧间等待 (16ms)，约占总帧预算的 3%。这意味着即使将渲染元素数乘以 10 倍（220 个元素），仍能在 60 FPS 目标内完成。这也是 P7（增量渲染）被推迟到「远期」的根本原因。

8.4.3 PHP↔C++ 调用开销

每帧 42 次 PHP↔C++ 调用通过 phpx 的类型转换层（详见 §4.7）。每次调用涉及：PHP 参数 → Variant 转换 → C++ 执行 → 返回值 Variant 装箱。对于简单的 GDI 原语（FillRect, TextOut），转换开销占比约 60-70%，实际 GDI 调用仅 30-40%。

优化方向：如果未来渲染成为瓶颈，可将 render() 中多次 GDI 调用合并为单次 C++ 批量调用（传递整个 elements/buttons 数组），减少 PHP↔C++ 往返次数。但当前场景不需要。
