## 简介与摘要

### 简介

本文档是 **VueCalc 项目** 从零到产出 `vue_calc.exe` 全过程的对话记录存档。项目目标是：基于 Swoole Compiler AOT 技术，使用 PHP 编写计算器业务逻辑 + C++ (Win32 GDI) 渲染界面，编译为 Windows 原生可执行文件。

开发过程历时多轮对话，跨越了两个会话（因上下文超限中断），经历了从架构设计、编译调试、运行崩溃排查到最终产出稳定 exe 的完整周期。

### 摘要

| 项目 | 内容 |
|------|------|
| **起始状态** | 框架代码已有骨架，但存在大量 AOT 不兼容问题，无法成功编译和运行 |
| **核心挑战** | Swoole Compiler AOT 对 PHP 语言的约束远超预期：不支持魔术方法、不支持扩展类、运行时函数白名单有限 |
| **最大 bug** | `__get/__set` 魔术方法在 AOT 编译后不被调用，导致属性读取返回 null，`$this->display .= $digit` 触发 TypeError 崩溃 |
| **关键转折** | 放弃基于魔术方法的响应式代理模式，改用**直接属性声明 + 手动脏标记** |
| **最终成果** | `vue_calc.exe` ~176KB，成功运行，支持四则运算、小数点、退格、清屏，双缓冲 GDI 渲染无闪烁 |
| **文档产出** | HTML 技术文档（41KB，含 Vue 对比 + SFC 编译器路线图）、Markdown 经验总结（343行）、本对话存档 |

### 技术亮点

- **架构演进**：从 Swoole 扩展依赖 → 纯 PHP → 魔术方法代理 → 直接属性（三次重构）
- **调试方法**：`no-console: false` + try-catch 全覆盖 + 编译验证循环
- **AOT 兼容模式**：显式声明 > 隐式惯例，编译期 > 运行时，手动 > 自动

## 第二阶段：Swoole 依赖剥离

### 2.1 Swoole 扩展不可用

**现象**：第一次成功编译的 exe 启动即闪退。

**原因**：原始代码使用了 `Swoole\Table` 和 `Swoole\Atomic` 类。AOT 编译的 exe 运行时不包含 Swoole 扩展，类找不到导致 fatal error。

**教训**：AOT 运行时 ≠ 标准 PHP CLI 运行时。编译产物只包含 PHP 核心运行时，不包含任何 PHP 扩展。

### 2.2 纯 PHP 重写响应式框架

**重写范围**：

| 原实现 | 新实现 |
|--------|--------|
| `ReactiveValue` 依赖 `Swoole\Table` | 纯 PHP 对象 `$value + $version` |
| `ChangeQueue` 依赖 `Swoole\Table` + `Swoole\Atomic` | 数组环形缓冲 `$buffer[$idx]` + head/tail 指针 |
| `ReactiveComponent::initShared` 使用 `Swoole\Table` | 纯 PHP `new ChangeQueue()` |

编译成功，exe 能启动并显示窗口，但点击按钮闪退（详见第四阶段）。

## 第三阶段：PHP 8 类型系统冲突

### 3.1 类型化属性拒绝代理对象

**错误**：`Cannot assign ReactiveValue to property Calculator::$display of type string`

**原因**：`Calculator` 中声明了 `protected string $display`，但 `initReactiveProperties()` 通过反射将 `new ReactiveValue('0')` 赋值给它。PHP 8 的类型化属性拒绝类型不匹配的赋值。

### 3.2 动态属性方案

**解决**：从 `Calculator` 中删除所有类型化属性声明，改用 `reactiveProperties()` 方法返回属性名→默认值数组。`ReactiveComponent` 通过 `__get/__set` 拦截所有未声明属性的读写，内部维护 `$proxies` 代理数组。

```php
// 不声明实际属性
class Calculator extends ReactiveComponent {
    protected function reactiveProperties(): array {
        return ['display' => '0', 'expression' => '', ...];
    }
}
// 所有读写经过 __get/__set → $proxies 数组 → ReactiveValue
```

这个方案在**标准 PHP** 中完美工作——直到 AOT 编译后又出问题（见第四阶段）。

## 第四阶段：点击闪退——最长调试战役

### 4.1 第一轮：添加诊断代码

**用户报告**："但是我点击数字按钮就闪退了！"

**诊断措施**：
- 修改 `project.yml`：`no-console: false`（从 `true` 改为 `false`）以显示错误输出
- 在 `handleClick()` 和 `render()` 外围添加 `try-catch(\Throwable)`
- 移除 `echo "Button: ..."` 避免 GUI 模式问题
- 编译运行：窗口能启动，但点击仍然崩溃

### 4.2 第二轮：修复 `any()`、`str_contains()`、`json_encode`

**用户提供的错误输出**：
```
ERROR: Cannot append element to an `null`
#0 [internal function]: Calculator->handleButton()
```

**初步分析**：错误发生在 `Calculator->handleButton()` 内部。审查代码后发现三个潜在 bug：

| Bug | 位置 | 说明 |
|-----|------|------|
| `any($value)` | `ChangeQueue.php:30` | **不是 PHP 函数**——致命错误 |
| `str_contains()` | `Calculator.php:124` | PHP 8.0 函数，AOT 可能不支持 |
| `json_encode`/`json_decode` | `ChangeQueue.php` | AOT 可能不可用 |

**修复**：
- `ChangeQueue::push()`：移除 `json_encode(any($value))`，改为直接存储字符串
- `Calculator::calculate()`：`str_contains()` → `strpos() !== false`
- `ReactiveComponent`：添加 `$queue` 空值检查

**结果**：错误从 `any()` 致命错误变成了更具体的 `Cannot append element to an null`。这说明 `any()` 确实是问题之一，但还有更深层的问题。

### 4.3 第三轮：`__get/__set` 在 AOT 中不工作（根因定位）

**错误分析**：

`Cannot append element to an null` 指向 `$this->display .= $digit` 这行代码。在标准 PHP 中，展开为：

```php
// 标准 PHP 执行
$temp = $this->display;              // → __get('display') → '7'
$this->display = $temp . $digit;     // → __set('display', '75')
```

但 AOT 编译后，`__get` 没有被调用，`$this->display` 返回 `null`，然后 `null .= '5'` 触发 TypeError。

**根因确认**：**Swoole Compiler AOT 不支持 `__get/__set` 魔术方法的自动调用**。这是 AOT 编译器将 PHP 翻译为 C++ 时的固有限制——未声明的属性在 C++ 层面不存在，编译器不会插入魔术方法调用。

### 4.4 最终方案：直接属性 + 手动脏标记

**架构变更**：

```
之前（不可行）：
  Calculator → reactiveProperties() → ReactiveComponent.__get/__set → ReactiveValue 代理

现在（AOT 兼容）：
  Calculator → public 属性声明 → 方法内部手动 $this->dirty = true → 主循环渲染
```

**具体修改**：

1. **ReactiveComponent.php**：精简到 30 行，仅保留 `$dirty`、`$queue`、`$componentId`、`$template` 和 `initShared()`
2. **Calculator.php**：声明 6 个 `public` 属性，每个修改状态的方法末尾加 `$this->dirty = true`
3. **不再使用** `__get/__set`、`ReactiveValue`、`reactiveProperties()`、`$proxies` 代理数组

**编译运行**：成功。点击数字按钮不再崩溃。

## 第五阶段：事件循环审查

### 5.1 用户发现时序问题

**用户指出**：如果 `$msgType == WM_LBUTTONDOWN` 一直有消息，就永远走不到渲染步骤。

**我最初的回应**：将渲染检查从外层移到内层（每条消息后立即渲染）。

### 5.2 C++ 源码确认消息机制

**用户再次审查**：`count($msg) == 0` 判断——如果消息队列中有多条消息，`vue_peek_message()` 是返回多条还是单条？

**阅读 C++ 源码**：

```cpp
// vue_calc.cc:70-84
Array php_vue_peek_message() {
    MSG msg;
    if (PeekMessage(&msg, NULL, 0, 0, PM_REMOVE)) {
        Array result;
        result.append((Int)msg.hwnd);       // [0]
        result.append((Int)msg.message);    // [1]
        result.append((Int)msg.wParam);     // [2]
        result.append((Int)msg.lParam);     // [3]
        return result;                       // 返回1条消息的4元素数组
    }
    return Array();                          // 无消息 → 空数组
}
```

**结论**：`vue_peek_message()` 每次返回 **1 条** 消息。内层 `while(true)` 循环逐条取出，`count($msg) == 0` 时表示队列已空。原始架构（内层清空消息→外层批量渲染）是正确的。

### 5.3 还原不必要的改动

将事件循环恢复为原始结构：内层处理所有消息 → 外层一次性检查 dirty → 渲染。用户确认这个改动没有必要。
